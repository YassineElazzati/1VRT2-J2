Les 4 modes réseau
de VirtualBox

Explications Métaphores filées Mini-TP par mode

Vue d'ensemble & métaphore parapluie

Une VM, c'est un ordinateur — donc elle a besoin d'une carte réseau. Dans VirtualBox, cette carte n'est pas physique : c'est un logiciel qui imite une vraie carte. Et ce logiciel peut être branché de 4 façons différentes au réseau du monde extérieur. Ces 4 façons s'appellent les modes réseau.

Le choix du mode change radicalement qui peut parler à qui : la VM voit-elle Internet ? Le PC physique peut-il joindre la VM ? Deux VM peuvent-elles dialoguer entre elles ? Tout dépend du mode choisi.

4 analogies simples à garder en tête. Pas besoin de comprendre un système global — chaque mode a sa propre image, tirée du quotidien :

  • NAT Le facteur. Votre courrier sort et entre par lui. Quand vous écrivez à quelqu'un, il met votre adresse à lui, pas la vôtre. Personne ne sait où vous habitez vraiment.
  • BRIDGE Un nouveau voisin dans le quartier. Il a sa propre maison, sa propre boîte aux lettres, sa propre adresse. Tout le monde peut lui écrire directement.
  • HOST-ONLY Un téléphone fixe entre votre maison et la cabane du jardin. Vous parlez avec ceux qui sont dans la cabane, eux parlent entre eux, mais ce téléphone n'appelle pas vers l'extérieur.
  • INTERNAL Des talkies-walkies entre enfants. Ils discutent entre eux sur leur canal. Les parents n'entendent rien, et les ondes ne passent pas les murs de la maison.

Avant de commencer, démarrez votre VM Ubuntu du Jour 1. Elle est en mode NAT par défaut — c'est notre point de départ. À chaque mini-TP, on changera son mode dans Configuration → Réseau → Carte 1 → Mode d'accès réseau.

NATLe mode par défaut

2.1 — Ce que c'est

NAT signifie Network Address Translation, ou « traduction d'adresses réseau » en français. Dans ce mode, VirtualBox joue le rôle d'un petit routeur invisible placé entre votre PC et la VM.

Conséquences pratiques :

  • La VM peut sortir vers Internet en passant par la connexion de votre PC
  • De l'extérieur, personne ne voit la VM — elle est cachée derrière votre PC
  • Votre PC lui-même ne peut pas joindre directement la VM (sans configuration spéciale)
  • Deux VM en NAT ne peuvent pas dialoguer entre elles, même côte à côte

La VM reçoit toujours la même IP factice : 10.0.2.15. Ce n'est pas une vraie IP du réseau — c'est juste celle que VirtualBox affiche à la VM. Pour le monde extérieur, le trafic semble venir de votre PC.

Le facteur. Imaginez que votre PC est le seul à avoir une boîte aux lettres sur la rue. Vos VM, elles, sont dans la maison sans boîte. Quand une VM veut écrire à quelqu'un (Google, Wikipédia…), elle confie son courrier au facteur (NAT). Le facteur enlève le nom de la VM, met votre adresse sur l'enveloppe, et l'envoie. Quand la réponse arrive, le facteur la transmet à la bonne VM. Résultat : pour le monde extérieur, c'est vous qui écrivez, jamais la VM. Et un livreur qui débarque de l'extérieur en demandant « VM1 » ? Il n'a aucune adresse à laquelle aller — la VM n'existe pas dehors.

INTERNET ▲ │ ┌──┴──────────────────────┐ │ PC HÔTE (192.168.1.10) │ │ │ │ ┌─────────────────┐ │ │ │ NAT (facteur) │ │ │ └────┬───────┬────┘ │ │ │ │ │ │ ┌────▼──┐ ┌──▼────┐ │ │ │ VM1 │ │ VM2 │ │ │ │10.0.2.│ │10.0.2.│ │ │ │ .15 │ │ .15 │ │ │ └───────┘ └───────┘ │ └─────────────────────────┘ VM → Internet OK PC hôte → VM BLOQUE (sauf redirection) VM1 → VM2 BLOQUE

À quoi ça sert dans la vraie vie :

  • Vous installez Ubuntu dans une VM pour tester un projet personnel — vous voulez juste Internet, pas que votre VM apparaisse sur le réseau de la maison ou de l'école.
  • Vous voulez télécharger des paquets (apt update) dans la VM sans la rendre visible.
  • Configuration par défaut sécurisée : la VM est invisible donc difficile à attaquer.

Mini-TP 1 — Vérifier qu'on est bien en NAT

Votre VM tourne déjà en NAT par défaut. On va le confirmer en 3 commandes.

STEP 01

Voir son IP dans la VM

ip addr show

Cherchez la ligne inet 10.0.2.15/24. Ce 10.0.2.15 est le marqueur du mode NAT — c'est toujours cette IP que VirtualBox attribue.

STEP 02

Tester Internet

ping -c 3 google.com

Ça doit répondre. La VM voit bien Internet, par l'intermédiaire du facteur.

STEP 03

Tenter de joindre la VM depuis le PC hôte

Sur votre PC physique (PowerShell ou Terminal), tapez :

ping 10.0.2.15

Ça ne répondra pas. C'est normal et c'est exactement ce que NAT fait : la VM est cachée. Pour la joindre depuis l'hôte en mode NAT, il faudrait configurer une redirection de port (port forwarding).

Bonus : la redirection de port. Si vous lancez un serveur web dans la VM (port 80) et que vous voulez y accéder depuis votre PC, allez dans Configuration → Réseau → Avancé → Redirection de ports, et ajoutez la règle : Port hôte 8080 → Port invité 80. Ensuite, ouvrez http://localhost:8080 dans votre navigateur. Vous donnez au facteur une consigne précise : « tout colis adressé à mon port 8080, dépose-le à la VM sur le port 80 ».

BRIDGELa VM sur le réseau local

3.1 — Ce que c'est

En mode Bridge (« pont » en français), votre carte réseau virtuelle se branche directement sur la carte physique de votre PC. La VM devient un ordinateur à part entière sur votre réseau local, comme si vous aviez branché un deuxième PC à côté du vôtre.

  • La VM reçoit une IP du routeur local (typiquement 192.168.1.x ou 192.168.0.x)
  • Elle est visible depuis tous les appareils du réseau (PC, smartphone, autre VM en bridge…)
  • Tout le monde sur le réseau peut la contacter directement
  • Elle a un accès Internet normal via le routeur

Le nouveau voisin dans le quartier. Un voisin emménage dans la maison à côté. Il a sa propre maison, sa propre boîte aux lettres, sa propre adresse. Le facteur (le routeur) la note dans son carnet. Désormais, tout le monde dans la rue peut lui écrire directement, sans passer par vous. Vous-même pouvez sonner à sa porte. Il fait partie du quartier comme n'importe qui, à part entière.

INTERNET ▲ │ ┌──┴──────────┐ │ ROUTEUR │ │192.168.1.1 │ └──┬──────┬──┘ │ │ │ └──────────┐ ▼ ▼ ┌────────────┐ ┌────────────┐ │ PC HÔTE │ │ VM Ubuntu │ │192.168.1.10│ │192.168.1.50│ └────────────┘ └────────────┘ VM → Internet OK PC hôte → VM OK (ping direct) Autre PC → VM OK (n'importe qui sur le LAN)

À quoi ça sert :

  • Vous montez un serveur web dans la VM et vous voulez que les autres machines du réseau (collègues, smartphone, autres PC) puissent ouvrir le site.
  • Vous testez une application client-serveur entre plusieurs machines.
  • Vous voulez vous connecter en SSH à votre VM depuis un autre poste du réseau.
  • Simulation d'un serveur d'entreprise : la VM se comporte comme un vrai serveur du LAN.

Sécurité. En Bridge, votre VM est exposée à tout le réseau. Si quelqu'un sur le même WiFi y trouve une faille, il peut entrer. Toujours avoir des mots de passe forts et, en production, un pare-feu actif.

Mini-TP 2 — Mettre la VM en Bridge et la rendre accessible

STEP 01

Changer le mode dans VirtualBox

  1. Éteignez la VM : sudo shutdown -h now
  2. VirtualBox Manager → sélectionner la VM → Configuration
  3. Réseau → Carte 1 → Mode d'accès réseau : Accès par pont
  4. Dans Nom, choisissez votre carte réseau active (WiFi ou Ethernet selon votre PC)
  5. OK, puis démarrez la VM
STEP 02

Vérifier la nouvelle IP

ip addr show
# Cherchez l'IP dans la plage 192.168.x.x ou 10.x.x.x
# C'est l'IP du LAN, pas plus 10.0.2.15

hostname -I
# Notez l'IP (ex: 192.168.1.50)
STEP 03

Pinguer la VM depuis le PC hôte

Sur le PC physique :

ping 192.168.1.50
# Remplacer par l'IP réelle relevée à l'étape 02
# Ça doit répondre cette fois
STEP 04

Installer un serveur web et le visiter

# Installer Apache
sudo apt update
sudo apt install -y apache2

# Vérifier qu'il tourne
sudo systemctl status apache2

Sur le PC hôte, ouvrez un navigateur et tapez : http://192.168.1.50

Vous devez voir la page « Apache2 Ubuntu Default Page ». Votre VM est désormais un vrai serveur web visible sur le réseau.

Demandez à un camarade du même réseau de taper aussi http://votre-ip-vm dans son navigateur. Il verra la même page. Vous venez d'héberger un site, depuis votre PC, accessible par tout le monde sur le LAN.

HOST-ONLYLe canal privé hôte-VM

4.1 — Ce que c'est

Le mode Host-only (« uniquement hôte ») crée un petit réseau virtuel privé qui n'existe nulle part ailleurs que sur votre PC. Seuls votre PC et les VM connectées à ce réseau peuvent y dialoguer.

  • Le PC hôte et les VM peuvent se parler librement
  • Les VM en Host-only peuvent aussi dialoguer entre elles
  • Mais aucun accès à Internet par cette carte
  • Le LAN extérieur (autres ordinateurs, smartphone) ne voit rien du tout

Le réseau créé porte typiquement le nom vboxnet0 avec une IP de type 192.168.56.1 côté hôte. Les VM reçoivent des IP dans cette plage (192.168.56.x).

Le téléphone fixe entre la maison et la cabane du jardin. Vous avez tiré un câble depuis chez vous (le PC hôte) jusqu'à une cabane au fond du jardin où jouent vos enfants (les VM). Vous pouvez appeler la cabane, les enfants peuvent vous appeler, et plusieurs enfants dans la cabane peuvent se parler. Mais ce téléphone ne sort pas du jardin : impossible d'appeler un numéro extérieur, et personne dehors ne peut composer ce numéro. C'est une ligne entièrement à vous.

RÉSEAU ISOLÉ (192.168.56.0/24) ┌────────────────┐ │ PC HÔTE │ │ 192.168.56.1 │ └────────┬───────┘ │ ┌─────┴─────┐ │ │ ┌───▼────┐ ┌───▼────-------┐ │ VM1 │ │ VM2 │ │192.168.56.10 │ 192.168.56.11│ └────────┘ └────────-------┘ PC ↔ VM OK VM ↔ VM OK VM → Internet BLOQUE LAN → VM INVISIBLE

À quoi ça sert :

  • Analyser un malware en toute sécurité : la VM infectée ne peut pas se propager sur Internet, mais le chercheur peut l'observer depuis l'hôte. C'est ce qu'on appelle un sandbox.
  • Connexion SSH stable à la VM, même si vous changez de réseau WiFi (le réseau Host-only ne dépend pas du WiFi).
  • Laboratoire de tests où plusieurs VM s'échangent des données sans risquer de polluer le réseau de l'école ou de l'entreprise.

Mini-TP 3 — Créer un réseau Host-only et s'y connecter en SSH

STEP 01

Créer le réseau Host-only

  1. VirtualBox Manager → Fichier → Outils → Gestionnaire de réseau hôte
  2. Onglet Host-only Networks → bouton Créer
  3. Un réseau vboxnet0 apparaît avec l'IP 192.168.56.1
  4. Vérifier que Serveur DHCP est activé (sinon, l'activer dans l'onglet DHCP Server)
  5. Appliquer
STEP 02

Ajouter une 2ᵉ carte réseau à la VM

On garde la Carte 1 pour Internet (NAT ou Bridge), et on ajoute une Carte 2 dédiée à Host-only.

  1. Éteindre la VM
  2. Configuration → Réseau → Carte 2
  3. Cocher Activer la carte réseau
  4. Mode d'accès réseau : Réseau privé hôte
  5. Nom : vboxnet0
  6. OK, démarrer la VM
STEP 03

Vérifier les interfaces dans la VM

ip addr show
# Vous devez voir 3 interfaces :
#  - lo       (loopback, 127.0.0.1)
#  - enp0s3   (Carte 1, NAT ou Bridge)
#  - enp0s8   (Carte 2, Host-only, IP 192.168.56.X)

# Si la 3ème n'a pas d'IP, forcer le rafraîchissement :
sudo systemctl restart systemd-networkd
STEP 04

Se connecter en SSH depuis l'hôte

Sur le PC physique (PowerShell ou Terminal) :

# Remplacer X par le dernier chiffre de l'IP Host-only de la VM
ssh votre_user@192.168.56.X

# Vous arrivez dans la VM, dans une fenêtre de terminal native
# de votre PC, avec copier-coller, redimensionnement, etc.

Pourquoi c'est génial. Vous avez maintenant deux portes d'entrée vers votre VM : la Carte 1 (Internet pour télécharger des paquets) et la Carte 2 (canal d'administration stable). Si le WiFi de l'école change, la Carte 2 continue de fonctionner. C'est exactement le pattern « réseau de management » qu'on trouve dans les vrais datacenters.

INTERNALLe huis-clos entre VM

5.1 — Ce que c'est

Le mode Internal Network (« réseau interne ») crée un réseau virtuel encore plus fermé que Host-only : même le PC hôte n'y a pas accès. Seules les VM connectées à ce réseau peuvent y dialoguer entre elles.

  • Les VM partageant le même réseau interne peuvent se parler
  • Le PC hôte est exclu de la conversation
  • Pas d'Internet, pas de LAN, pas d'accès extérieur
  • C'est un cocon totalement étanche

On désigne le réseau par un nom qu'on choisit librement (ex. backend-network). Toutes les VM qui partagent ce nom sont sur le même réseau interne ; celles qui ont un nom différent ne se voient pas.

Les talkies-walkies entre enfants. Vos enfants jouent dans le jardin avec des talkies-walkies sur la même fréquence. Ils discutent entre eux librement. Vous, dans la maison, vous n'entendez rien — vous n'avez pas de talkie. Et un voisin un peu plus loin ? Les ondes ne portent pas jusqu'à chez lui. Les enfants sont entre eux, point. Personne d'autre dans la conversation, pas même les parents.

RÉSEAU INTERNE (nom: backend-network) 10.0.10.0/24 (à configurer manuellement) ┌────────────────┐ │ VM Web │ │ 10.0.10.10 │ └────────┬───────┘ │ ┌─────┴─────┐ │ │ ┌───▼────┐ ┌───▼────┐ │ VM DB │ │VM File │ │10.0.10.│ │10.0.10.│ │ .20 │ │ .30 │ └────────┘ └────────┘ VM ↔ VM (même réseau) OK PC hôte → VM INVISIBLE VM → Internet BLOQUE LAN extérieur INVISIBLE

À quoi ça sert :

  • Sécuriser une base de données qui ne doit être accessible que par les serveurs web — pas par les utilisateurs. La base reste sur un réseau interne ; seul le serveur web a une jambe sur le réseau interne et une jambe sur Internet.
  • Simuler une infrastructure d'entreprise avec un réseau « front » exposé et un réseau « back » caché. C'est exactement le projet final TP6.
  • Tests de protocoles réseau en environnement totalement contrôlé.

Mini-TP 4 — Deux VM qui se parlent en huis-clos

Pour ce mini-TP, vous avez besoin de deux VM. Si vous n'en avez qu'une, créez d'abord un clone (clic droit sur la VM → Cloner, n'oubliez pas de cocher « Générer de nouvelles adresses MAC »).

STEP 01

Configurer les deux VM en Internal Network

  1. Éteindre les deux VM
  2. Pour chacune : Configuration → Réseau → Carte 2
  3. Cocher Activer la carte réseau
  4. Mode d'accès réseau : Réseau interne
  5. Nom : backend-network (taper le même nom des deux côtés !)
  6. OK pour les deux VM, puis démarrer
STEP 02

Attribuer des IP manuellement

Le mode Internal Network n'a pas de serveur DHCP par défaut. Il faut donner une IP fixe à chaque VM.

Sur la VM 1 :

# Identifier l'interface Internal (souvent enp0s8)
ip addr show

# Lui attribuer l'IP 10.0.10.10 temporairement
sudo ip addr add 10.0.10.10/24 dev enp0s8
sudo ip link set enp0s8 up

Sur la VM 2 :

sudo ip addr add 10.0.10.20/24 dev enp0s8
sudo ip link set enp0s8 up
STEP 03

Tester la communication

Depuis la VM 1, pinguer la VM 2 :

ping -c 3 10.0.10.20
# Doit répondre — les deux VM dialoguent en huis-clos
STEP 04

Vérifier que l'hôte est exclu

Sur le PC hôte :

ping 10.0.10.10
# Pas de réponse — l'hôte est sur le palier, la salle est fermée

Vous pouvez aussi vérifier depuis la VM qu'elle n'a pas d'Internet via cette carte :

# Désactiver temporairement la Carte 1 pour le test
sudo ip link set enp0s3 down
ping -c 2 google.com
# "Network unreachable" — c'est attendu

# Réactiver Internet
sudo ip link set enp0s3 up

Les commandes ip addr add sont temporaires : elles disparaissent au redémarrage. Pour rendre la config permanente, on utilise netplan — c'est ce qu'on fera dans le projet final TP6 cet après-midi.

Le quizz à se faire à soi-même. Pour chaque mode, demandez-vous trois choses :

  1. La VM a-t-elle accès à Internet ?
  2. Le PC hôte peut-il joindre la VM ?
  3. Une autre VM peut-elle joindre celle-ci ?

Si vous savez répondre par OUI/NON pour chaque mode, vous avez tout retenu.

Pièges classiques

« Toutes les VM ont l'IP 10.0.2.15 en NAT ? » Oui, et ce n'est pas un bug. En NAT, chaque VM est isolée dans son propre réseau privé. Aucune ne voit les autres, donc aucune ne sait qu'elles ont la même IP. Le facteur (VirtualBox) gère les conflits en coulisses.

« J'ai mis Internal Network mais ma VM n'a pas d'IP. » Normal : Internal Network n'a pas de DHCP par défaut. Soit vous attribuez les IP manuellement (ip addr add ou netplan), soit vous activez le DHCP avec VBoxManage dhcpserver add.

« Mes deux VM en Internal ne se voient pas. » Vérifiez que le nom du réseau interne est strictement identique sur les deux VM (espace, casse, tout compte). Si l'une est sur backend-network et l'autre sur backend, elles sont sur deux réseaux différents et invisibles l'un de l'autre.

« Bridge ne marche pas chez moi. » Les WiFi d'entreprise ou d'école bloquent souvent le mode Bridge pour des raisons de sécurité (un même appareil ne doit pas multiplier ses identités). Solution : tester sur un réseau personnel, ou utiliser Host-only + redirection de ports en NAT.

« Host-only ou Internal ? » Confusion fréquente. Repère simple : si le PC hôte doit participer à la conversation → Host-only. Si seules les VM doivent dialoguer (et que l'hôte ne doit rien voir) → Internal.

Combiner plusieurs cartes : le pattern à connaître. Une VM peut avoir jusqu'à 4 cartes réseau, chacune dans un mode différent. C'est ce qu'on appelle le multi-homing. Le combo le plus utile : Carte 1 en NAT (Internet) + Carte 2 en Host-only ou Internal (administration et communication interne). C'est exactement la config qu'on déploiera dans le projet final cet après-midi.
