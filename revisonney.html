Fondamentaux de la virtualisation

NotionDéfinition
VirtualisationTechnologie permettant de créer des versions virtuelles de ressources physiques (serveurs, stockage, réseau).
Host (Hôte)La machine physique qui fournit les ressources matérielles réelles (CPU, RAM, disque).
Guest (Invité)La machine virtuelle (VM) qui s'exécute sur l'hôte avec son propre OS.
HyperviseurCouche logicielle qui crée, gère et isole les VM, et répartit les ressources du Host entre elles.
IsolationChaque VM est indépendante. Si une VM plante, les autres continuent normalement.

Ce que chaque VM possède en propre

  • Son propre système d'exploitation complet et indépendant
  • Ses propres applications
  • Ses ressources virtuelles : vCPU, vRAM, disque virtuel (.vdi), carte réseau virtuelle

Avantages clés

AvantageDétail / Chiffre
Optimisation matériel10-20 % → 70-80 % d'utilisation des ressources
Réduction coûts50-70 % d'économies (matériel, énergie, maintenance)
Empreinte carboneMoins de serveurs physiques = moins d'électricité et de climatisation
IsolationUn crash de VM n'affecte pas les autres
Adoption marché~90 % des serveurs d'entreprise virtualisés (Gartner)

Faux mais piégeant : « la virtualisation élimine le besoin de matériel physique » (non, elle l'optimise) ; « rend les serveurs invulnérables aux pannes » (non) ; « supprime le besoin d'OS dans les VM » (non, chaque VM a son OS).

Astuce mémo : Type 1 = touche le métal (bare-metal). Type 2 = a un OS en dessous (hosted). VirtualBox sur votre PC ? Il y a Windows dessous → Type 2.

Installation et vérification VirtualBox

Vérifier la virtualisation matérielle

OSCommandeSortie attendue
Windowssysteminfo | findstr /i "virtualization"Lignes contenant « Virtualization Enabled In Firmware: Yes »
Linuxegrep -c '(vmx|svm)' /proc/cpuinfoNombre > 0 (= nombre de threads supportant VT-x/AMD-V)

Piège ULTIME : ces commandes LISENT seulement. Elles n'activent rien, n'installent rien, ne désactivent Hyper-V de rien. Aucune commande OS ne modifie le BIOS — pour activer VT-x, il faut entrer dans le BIOS au démarrage.

VBoxManage et Extension Pack

ÉlémentDétail
VBoxManage --versionAffiche la version installée (en lecture seule, n'installe rien)
VirtualBox Extension PackAjoute : USB 2.0/3.0, chiffrement AES, boot PXE Intel, accès RDP distant. Gratuit pour usage éducatif.
sudo usermod -aG vboxusers $USERAjoute l'utilisateur au groupe vboxusers (USB). Le flag -a = append (sans lui, on remplace tous les groupes).

Le piège -aG : sans le -a, usermod -G vboxusers $USER remplacerait tous vos groupes par vboxusers seulement. On perdrait sudo, audio, video, etc. Le -a = append = ajouter sans retirer.

Dimensionnement d'une VM

RAM

Maximum 50-60 % de la RAM totale aux VM.

Ex. PC 8 GB → max 4-5 GB aux VM.

Garder 40-50 % pour l'OS hôte et l'hyperviseur.

CPU

Maximum 50-75 % des cœurs aux VM.

Ex. 4 cœurs → max 2-3 vCPU aux VM.

La virtualisation ne crée pas de cœurs — elle les partage.

Pièges : « 100 % RAM aux VM » = faux (crash hôte). « 4 cœurs aux VM sur 4 » = faux (hôte étouffé). « Virtualisation double le nombre de cœurs » = faux (elle les partage, ne les crée pas).

Question type : « Quel mode pour analyser un malware ? » → Host-only (pas d'Internet pour la propagation, mais observation possible depuis l'hôte).

Pattern entreprise : double carte réseau = Bridge (trafic applicatif/Internet) + Host-only (management SSH stable). Le Host-only ne dépend pas du réseau externe — utile si le WiFi change.

Accès web à une VM en NAT depuis l'hôte : il faut une redirection de port (ex. port hôte 8080 → port invité 80). L'IP 10.0.2.15 n'est pas joignable directement.

Commandes réseau Ubuntu

CommandeRôle
ip addr showAffiche toutes les interfaces réseau et leurs IP (lecture seule)
sudo systemctl restart systemd-networkdRedémarre le service réseau (force un nouveau DHCP request)
sudo apt install -y apache2Installe Apache. -y = accepter la confirmation auto
sudo systemctl status apache2Vérifie l'état du service (active / running ?)
sudo ufw allow 80/tcpAutorise les connexions entrantes port 80 TCP (HTTP) dans UFW
sudo netplan applyApplique la configuration réseau définie dans /etc/netplan/*.yaml

Pièges classiques : apt install installe (ne met pas à jour, c'est apt upgrade). systemctl status est en lecture seule (n'arrête, ne redémarre rien). ufw allow ouvre un port (le bloquer, c'est ufw deny).

Snapshots

NotionDéfinition
SnapshotSauvegarde instantanée de l'état complet d'une VM : disque + RAM (si la VM tourne) + configuration
Snapshot vs SauvegardeSnapshot = incrémental, rapide, quasi instantané. Sauvegarde complète = copie entière du disque, lent, plus volumineux
Quand l'utiliser ?Avant une mise à jour ou modification risquée (rollback instantané possible)
Pourquoi nettoyer ?Les snapshots consomment de l'espace disque croissant et peuvent ralentir les I/O de la VM

Confusions à éviter : un snapshot n'est pas une capture d'écran, n'est pas une copie indépendante (c'est lié à la VM), n'est pas un rapport de performance.

Astuce TP : créer un fichier témoin avant snapshot avec echo "État initial" > ~/test-snapshot.txt pour vérifier ensuite que la restauration a bien fonctionné.

Clonage de VM

Type de cloneCaractéristique
Clone completCopie totalement indépendante. Autonome, portable, mais volumineux.
Clone liéPartage le disque de base de l'original. Stocke uniquement les différences. Léger mais dépendant.

Règle d'or du clonage : toujours générer de nouvelles adresses MAC lors du clonage. Deux MAC identiques sur le même réseau = conflit. C'est une obligation, pas une suggestion.

Renommer le hostname du clone (sinon deux machines portent le même nom sur le réseau) :

sudo hostnamectl set-hostname ubuntu-clone

Modification permanente, immédiate. Géré par systemd.

Migration à froid vs migration à chaud

MigrationComportementOutil
À froidVM éteinte avant transfert. Interruption de service.Export/import manuel
À chaudVM en cours d'exécution, transfert progressif de la RAM. Interruption de quelques millisecondes.vMotion (VMware), Live Migration (Proxmox/KVM)

Haute Disponibilité (HA)

ComposantRôle
Cluster HAPlusieurs serveurs physiques qui partagent les VM et se surveillent mutuellement
HeartbeatSignal régulier échangé entre serveurs pour détecter une panne en quelques secondes
Stockage partagé (SAN)Composant clé du HA. Tous les serveurs y accèdent simultanément. Sans SAN, pas de redémarrage automatique possible.
Comportement en panneLes VM du serveur en panne sont automatiquement redémarrées sur les autres serveurs du cluster

Pièges HA : les VM ne sont pas perdues (le SAN les conserve). Toutes les VM du cluster ne s'arrêtent pas (seules celles du serveur en panne migrent). Pas d'intervention humaine nécessaire (c'est automatisé).

Top pièges de big GOLMON Tiépo

1. Une commande de vérification (systeminfo, egrep cpuinfo, VBoxManage --version, systemctl status, ip addr show) ne modifie jamais rien. Si une option dit qu'elle « installe », « active », « désactive » ou « supprime » : c'est faux.

2. Aucune commande depuis l'OS ne peut activer la virtualisation dans le BIOS. L'activation se fait uniquement au démarrage du PC, dans le BIOS/UEFI.

3. Type 0 et Type 3 n'existent pas. Il n'y a que Type 1 (bare-metal) et Type 2 (hosted).

4. Tout pourcentage à 100 % dans un QCM de virtualisation est faux. On garde toujours une marge pour l'hôte.

5. Le format VDI est le format natif de VirtualBox. VMDK = VMware. VHD = Hyper-V. ISO = image CD/DVD (PAS un disque dur virtuel).

6. Pour accéder à une VM en NAT depuis l'hôte → redirection de port. Pas via 10.0.2.15 directement, pas en désactivant le pare-feu.

7. Pour analyser un malware → Host-only (pas Internal, pas NAT, jamais Bridge).

8. Un snapshot capture l'état complet (disque + RAM + config). Pas une capture d'écran, pas une copie indépendante, pas un rapport de perf.

9. Lors d'un clonage, toujours générer de nouvelles adresses MAC. Sinon = conflit réseau garanti.

10. Dans un cluster HA : panne d'un serveur → les VM redémarrent automatiquement sur les autres serveurs grâce au SAN (stockage partagé) et au heartbeat (détection).

11. vMotion = migration à chaud. DRS = équilibrage automatique. Fault Tolerance = réplique synchrone. vCenter = console centralisée. Ne pas les confondre.

12. Proxmox est basé sur KVM (pas Docker, pas ESXi, pas Xen). KVM est intégré au kernel Linux.

13. Hyper-V est inclus dans Windows Server (gratuit pour qui a la licence Windows). Mais Hyper-V lui-même n'est pas open-source ni gratuit hors de Windows Server.

14. Le flag -a de usermod -aG est obligatoire. Sans lui, on perd tous les autres groupes de l'utilisateur.

15. En NAT, toutes les VM ont l'IP 10.0.2.15. Ce n'est pas un bug : chaque VM est dans son propre réseau privé, isolée des autres.

Le réflexe gagnant : avant de répondre, demandez-vous toujours (1) cette commande lit-elle ou modifie-t-elle l'état ? (2) le chiffre est-il dans la bonne fourchette ? (3) cette affirmation contient-elle un mot absolu (« toujours », « jamais », « 100 % », « éliminer », « supprimer ») ? Les mots absolus sont presque toujours faux.
